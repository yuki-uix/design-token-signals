# Product Positioning — design-token-signals

> 这份文档是产品的北极星。开发新功能前先看这里，确保每一个决策都在这个方向上。

---

## 我们在解决什么问题

**AI 生成 UI 的速度很快，但没有设计依据时，每次生成的结果是随机的，不成系统。**

具体表现：
- 同一个产品的不同页面，AI 生成的 UI 风格各自为政
- 开发者知道想要什么「感觉」，但说不清楚对应哪些参数
- 每次新 idea 都要重新跟 AI 描述风格，没有可复用的设计锚点

---

## 产品是什么

**design-token-signals 是面向 AI 辅助开发者的设计风格选型工具 + 设计系统学习站。**

两个入口，一个底层：

- **工具入口**：我有一个 idea，告诉我应该用什么设计系统参数，给我一个能放进 repo 的配置文件
- **Docs 入口**：我想理解设计系统是怎么运作的，这里有设计信号理论的学习笔记和拆解

底层是同一套东西：**signal → token → 配置文件** 这条链路的完整阐释。

---

## 目标用户

**主要：AI 辅助开发者（独立开发者 / 全栈工程师 / 设计工程师）**
- 用 Cursor / Claude Code 等工具快速构建产品
- 有 UI 一致性需求，但没有专职设计师
- 知道自己想要什么「气质」，但需要帮助把它翻译成设计参数

**次要：对设计系统感兴趣的学习者**
- 想理解为什么某些 token 组合「感觉对了」
- 想积累设计系统相关的知识

---

## 成功场景（北极星）

> 我要做一个 planning-poker 工具，想要商务办公但不至于太严肃。
> 我在 design-token-signals 搜索关键词，找到匹配的主题，下载配置文件，
> 放进 repo 里，之后 AI 生成的每个功能页面都有设计依据。

**这个场景成立的那一天，产品就做成了。**

---

## 产品不是什么

- ❌ 不是 Figma 插件或设计工具的替代品
- ❌ 不是提供完整 UI 组件库的资源站
- ❌ 不是帮 AI 「学会设计」的工具（那是 impeccable.style 做的事）
- ❌ 不是面向设计师的工具（主要用户是开发者）

---

## 三个 Epic

### Epic 1 · 选型工具（Tool）
*帮用户从「我有一个 vibe」到「我有一个可以用的配置文件」*

核心工作流：描述意图 → 找到匹配主题 → 下载 AI-ready 配置文件

| 状态 | 功能 |
|---|---|
| ✅ 完成 | 9 套主题 demo 展示 |
| ✅ 完成 | Signal meta 内容层（适用场景 / 排除场景 / rationale） |
| ✅ 完成 | CSS 变量导出 / Token JSON 导出 / URL 分享 |
| ✅ 完成 | AI-ready 配置文件导出（`.signal.md`，Cursor / Claude Code 可直接读）|
| 🔲 待做 | 关键词 / intent 匹配（搜索或引导问卷）|

**这是产品的核心差异化。关键词匹配是入口，配置文件是终态。**

### Epic 2 · 文档层（Docs）
*记录设计信号理论，沉淀个人学习积累*

核心价值：别的工具有 token，只有这里解释 token 背后的「为什么」

| 状态 | 功能 |
|---|---|
| ✅ 部分 | README 中的 signal 维度框架、/analysis、/conflicts 目录 |
| 🔲 待做 | Docs 板块（独立页面）|
| 🔲 待做 | Signal 理论文章（radius 影响亲和感、accent 饱和度影响信任感……）|
| 🔲 待做 | 每套主题的深度解读 |

### Epic 3 · 体验完整性（Polish）
*补完工具流程上的断点*

| 状态 | 功能 |
|---|---|
| ✅ 完成 | Legend 面板信息层级重设计（主 CTA 突出、字号可读性、宽度扩展）|
| 🔲 待做 | 完整 token 预览（legend 可展开查看 29 个变量）|
| 🔲 待做 | Demo 页上下文锚点（进入后知道自己在哪）|
| 🔲 待做 | 多格式导出（W3C Design Token 格式）|

## 当前 Roadmap

```
本周          →  S9: 导出后使用引导（.signal.md 下一步提示 + 团队用法说明）
                  目标：北星场景「用起来」的最后一公里，成本极低，优先清掉

下一个 sprint →  S6: 关键词/Intent 匹配
                  注意：先标注 3 套主题 keywords 内测，结果通过再全量推进
                  keywords 质量 = 功能成败，不是搜索逻辑本身

中期          →  Eleventy 迁移（Docs 板块开始前，不提前动）
              →  Epic 2: Docs 板块（先上线 1 篇，不等写满）
              →  Epic 3: Legend 常驻 Signal（等用户反馈再决定）

长期          →  更多主题 / 支持用户自定义 signal
```

## 暂不做（已决策）

| 方向 | 原因 |
|---|---|
| 多场景 Demo（landing/dashboard/form）| 回答了用户没有问的问题，维护成本 3 倍 |
| 跨主题对比模式 | 用例频率低，性价比存疑 |
| 主题数据 JSON 化（提前重构）| 随 Eleventy 迁移一起做，现在做等于做两遍 |
| W3C Design Token 格式导出 | 服务非核心用户群，稀释 AI native 定位 |

## 技术约束（保持意识）

- 纯静态站，GitHub Pages，无构建工具，无框架
- 不引入后端服务（关键词搜索在客户端完成）
- 配置文件导出是纯客户端生成，不需要服务器
